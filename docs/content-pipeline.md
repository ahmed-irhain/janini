# Content Pipeline: Articles, Supabase, and AI Generation

This doc covers two things that don't exist yet in `docs/plan.md`: (1) how the
existing article content pipeline moves onto a Supabase-hosted Postgres
database, and (2) how new articles get AI-generated — tone, non-negotiable
constraints, and the actual prompt design. Read `docs/plan.md` Section 6.2
first for *why* content is AI-drafted-and-cited rather than OB-GYN-reviewed;
this doc is the *how*.

## 1. Current state

The pipeline already exists and this doc doesn't change its shape:

- **Schema** (`server/src/db/schema.ts`): `articles` (title/summary/body,
  optional `weekNumber` FK to `weeklyContent`, optional `topicId`,
  `sourceName`/`sourceUrl`), `topics`, `weeklyContent` (with a
  `sourceCitations` array).
- **Content source of truth**: frontmatter `.md` files in
  `server/content/articles/` — see any file there for the exact format
  (`title:`/`summary:`/`sourceName:`/`sourceUrl:`/optional `weekNumber:`,
  then a `---`-delimited Markdown body).
- **Loader**: `server/src/db/importArticles.ts` (run via
  `npm run db:import-articles --workspace=server`) parses those files and
  upserts them by `titleAr`, matched against whichever `DATABASE_URL` is set.
- **Read API**: `server/src/routes/articles.ts` — `GET /articles/id/:id`,
  `GET /articles/:week`, `GET /articles?topic=...`.

Nothing below replaces any of this. Supabase changes *where* Postgres runs;
AI generation changes *how* the `.md` files get written. The loader and
schema stay exactly as they are.

## 2. Supabase workflow (hosted Postgres only)

**Scope note:** Supabase is used here purely as **hosted Postgres**. The
app's existing custom auth (`users` table, bcrypt password hashing in
`auth/password.ts`, JWT sessions in `auth/jwt.ts`) stays as-is. Supabase
Auth, Storage, Realtime, and Edge Functions are explicitly out of scope —
this is a deliberate choice, not an oversight, so it shouldn't get quietly
reconsidered later without a real decision to do so.

### 2.1 Why Drizzle stays even though Supabase has a dashboard

Supabase's Table Editor and SQL Editor are a GUI over Postgres — useful for
eyeballing rows or running an ad-hoc query, but not a schema-management or
application-integration tool. Three concrete reasons dropping Drizzle for
dashboard-only editing would break this app:

1. **Every route is a type-checked Drizzle query against `schema.ts`.**
   `server/src/routes/articles.ts`, `auth.ts`, etc. all call `db.select()`/
   `db.insert()` against the schema's TypeScript types. A schema change made
   through Supabase's dashboard is invisible to the compiler — the code
   silently drifts from the real database until something breaks at runtime.
2. **Every schema change is a reviewable, committed migration.** Each change
   so far is a `.sql` file under `server/drizzle/` (e.g.
   `0003_tidy_wendell_rand.sql`), generated by `drizzle-kit`, committed to
   git, and applied identically everywhere via `db:migrate`. Manual dashboard
   edits leave no history and let environments quietly diverge from each
   other and from what's in git.
3. **Multiple environments need the identical schema.** Local dev
   (docker-compose) and Supabase staging/prod all need the same table
   structure. `db:migrate` guarantees that by replaying the same versioned
   SQL against whichever `DATABASE_URL` it's pointed at — there's no
   equivalent "apply this everywhere" button in a dashboard.

Use Supabase Studio for convenience (inspecting data, debugging a query) —
never for making a schema change. Schema changes always go through
`drizzle-kit generate` → commit → `db:migrate`.

### 2.2 Connection strings: which of Supabase's three to use where

Supabase exposes three connection modes for the same database:

| Connection type | Use for | Why |
| --- | --- | --- |
| **Direct connection** | Running migrations (`db:migrate`) | Migrations need a plain, non-pooled session; pooled connections can behave unexpectedly with `drizzle-kit`'s migration transactions. |
| **Session pooler** | The long-lived Express server process (`npm run dev`/`start` in `/server`) | A persistent Node process holding a handful of connections is exactly what the session pooler is for — behaves like a normal Postgres connection, just pooled. |
| **Transaction pooler** (pgbouncer, port 6543) | Not needed today | Only relevant if the server ever moves to a serverless/edge deployment model, where connections are opened per-request. Comes with caveats (no prepared statements) that don't matter for a long-lived process. Flag as a future concern only, not something to configure now. |

### 2.3 Environments and secrets

One `DATABASE_URL` (and, where needed, a `DATABASE_URL_DIRECT` for
migrations) per environment: local docker-compose, Supabase staging project,
Supabase prod project. Each lives in its own `.env`, never committed —
`server/.env.example` already documents the shape (`DATABASE_URL`, `PORT`,
`JWT_SECRET`). No code branches on environment; only the connection string
differs. Production credentials belong in a secrets manager or CI secret
store, never in chat, never in a committed file.

### 2.4 Migration flow — same shape, new target

```bash
npm run db:generate --workspace=server   # writes SQL under server/drizzle/, commit it and review like any code change
npm run db:migrate --workspace=server    # applies it — point DATABASE_URL at whichever environment is the target
```

No Supabase-specific migration tooling is needed; Drizzle's migrator works
against any Postgres server, including Supabase's.

### 2.5 Content promotion path

The same loader that seeds dev today is the same loader used for
staging/prod — only the connection string changes:

```bash
DATABASE_URL=<target> npm run db:import-articles --workspace=server
```

Articles are just rows keyed by `titleAr` (upsert semantics — running the
import again updates existing articles rather than duplicating them). There
is no separate "publish to Supabase" step beyond running the import against
the right `DATABASE_URL`.

## 3. AI-generation workflow

### 3.1 Non-negotiable constraints (recap)

Every generated article must satisfy the rules already in root `CLAUDE.md`
and `docs/plan.md` §6.2 — this doc doesn't relax or duplicate them, just
operationalizes them into a prompt:

- **MSA only**, no dialect commitment (Gulf-conversational tone is reserved
  for onboarding/notifications/community copy, not medical content).
- **Zero diagnostic or treatment language.** Describe symptoms and direct to
  "consult your doctor" — never name or imply a diagnosis, never recommend a
  specific treatment.
- **Every factual claim must be grounded in and cite a named trusted
  authority** (WHO, Saudi MOH/SFDA, ACOG, Mayo Clinic, or a similarly
  credible institution — e.g. a Gulf-region hospital system for
  culturally-specific topics like Ramadan fasting). At least one
  `sourceUrl`/`sourceName` per article, and it must be a real, checkable URL
  — never a plausible-sounding but unverified one.
- **Gulf cultural awareness** where relevant (halal dietary notes, Ramadan
  fasting, the 40-day postpartum period) without committing to one country's
  dialect or customs.

### 3.2 Pipeline stages

1. **Backlog input** — a topic or week number plus 1–2 lines of intent (e.g.
   "general article on safe exercise during pregnancy, cite ACOG").
2. **Generation** — a Claude API call (model: `claude-opus-4-8`, current
   default) using the prompt template below.
3. **Human review** — the checklist in §3.4, before anything is saved.
4. **Save & import** — reviewed output saved as a `.md` file under
   `server/content/articles/`, committed, then loaded via the same
   `db:import-articles` script (see §2.5) against the target environment.
5. **Re-citation cadence** — per `docs/plan.md` §6.2, re-check citations
   periodically and whenever a cited guideline updates. This is a recurring
   process step, not a one-time gate.

### 3.3 Prompt template

**System prompt:**

```
You write medically-grounded articles in Modern Standard Arabic (Fusha) for
Janini, an Arabic-first pregnancy-tracking app for Saudi Arabia and the Gulf.

Non-negotiable rules:
- Write in MSA only. Do not use a specific country's dialect or slang.
- Never use diagnostic or treatment language. Describe symptoms and general
  guidance only; always direct the reader to consult her doctor rather than
  naming or implying a diagnosis or prescribing a treatment.
- Every factual claim must be grounded in one real, verifiable source from a
  recognized health authority: WHO, the Saudi Ministry of Health/SFDA, ACOG,
  Mayo Clinic, or an equivalently credible institution (a named Gulf-region
  hospital system is acceptable for culturally-specific topics). Do not
  fabricate a URL or source name — if you are not certain a source exists
  and says what you're citing it for, say so instead of inventing one.
- Where culturally relevant, reflect Gulf context (halal dietary guidance,
  Ramadan fasting considerations, the 40-day postpartum period) without
  committing to one country's customs.
- Output ONLY the following format, nothing else:

---
title: <Arabic title>
summary: <2-3 sentence Arabic summary, ending with a consult-your-doctor note
  if the topic touches anything symptom-related>
sourceName: <source institution name, Arabic if it has a common Arabic name,
  otherwise the English name>
sourceUrl: <the real source URL>
---
<Arabic Markdown body: use ## headings, bullet or numbered lists, and end
with a blockquote (`>`) reinforcing the consult-your-doctor framing where
relevant>
```

**User prompt (fill in per backlog item):**

```
Topic: <e.g. "safe exercise during pregnancy">
Target week (optional): <week number, or "general — not tied to a specific week">
Preferred source (optional): <e.g. "ACOG" — omit to let the model choose an
  appropriate authority>
Any additional context: <e.g. "this is for the general articles feed, not
  week-specific content">
```

This is written to be directly usable once a generation script exists (see
§3.6) — copy it in verbatim rather than re-deriving prompt language later.

### 3.4 Human review checklist

This checklist **is** the safety net now that there's no paid OB-GYN review
pass (`docs/plan.md` §6.2). Nothing gets saved or imported until it passes:

- [ ] At least one real, checkable `sourceUrl` is present, and it's accurate
      to the specific claims made in the article (not just topically related).
- [ ] No diagnostic or treatment language anywhere — scan for phrasing that
      names or implies a condition rather than describing a symptom plus
      "see your doctor."
- [ ] MSA register maintained throughout; no dialect slippage.
- [ ] Gulf-appropriate framing, not tied to one country's customs or slang.
- [ ] Frontmatter matches the exact format `importArticles.ts` parses
      (`title:`/`summary:`/`sourceName:`/`sourceUrl:`, optional `weekNumber:`,
      then the `---`-delimited body) — a malformed file throws at import time.

### 3.5 Save & import

Once reviewed:

```bash
# save the reviewed output as server/content/articles/<slug>.md, then:
git add server/content/articles/<slug>.md
git commit -m "Add article: <title>"
DATABASE_URL=<target> npm run db:import-articles --workspace=server
```

### 3.6 Next step (not built this round)

This doc is prompt design and process, not code. The natural next step is a
`server/src/scripts/generate-article.ts` that takes the backlog item as CLI
input, calls the Claude API with the system/user prompt above, and writes the
raw output to a draft `.md` file for human review (§3.4) before it's ever
committed or imported — deliberately not auto-saving straight into
`server/content/articles/`, since that's where the human review step lives.
